# JWT Refresh Token Rotation 트러블슈팅 정리

## 1. 문제 상황

JWT 인증 구조에서 다음과 같은 방식을 사용했다.

```txt
accessToken: 프론트 Zustand 메모리
refreshToken: HttpOnly Cookie
refresh token rotation 사용
axios interceptor에서 401 처리
```

단일 요청 기준으로는 정상 작동했다.

```txt
API 요청
→ 401
→ /auth/refresh
→ 새 accessToken 저장
→ 원래 요청 재시도
→ 성공
```

하지만 인증이 필요한 API 여러 개가 동시에 401을 받으면 세션이 꼬이는 문제가 발생했다.

---

## 2. 원인

refresh token rotation에서는 refresh token을 한 번 사용하면 새 refresh token으로 교체한다.

동시에 여러 요청이 401을 받으면 기존 코드에서는 이런 일이 생긴다.

```txt
/books  → 401 → /auth/refresh A
/users  → 401 → /auth/refresh B
/orders → 401 → /auth/refresh C
```

첫 번째 refresh는 성공한다.

```txt
refresh A 성공
→ 새 accessToken 발급
→ 새 refreshToken 발급
→ 기존 refreshToken 폐기
```

하지만 B, C도 거의 동시에 기존 refresh token으로 refresh를 시도한다.

```txt
refresh B
→ 이미 폐기된 refreshToken 사용
→ 실패

refresh C
→ 이미 폐기된 refreshToken 사용
→ 실패
```

따라서 refresh token rotation 환경에서는 **동시 refresh 요청이 경쟁 상태를 만들고 세션을 꼬이게 한다.**

---

## 3. 핵심 해결책

refresh 요청을 여러 번 보내지 않고, **진행 중인 refresh Promise를 공유**한다.

```ts
let refreshPromise: Promise<string> | null = null;
```

401이 발생했을 때:

```ts
if (!refreshPromise) {
  refreshPromise = api
    .post("/auth/refresh")
    .then((res) => res.data.accessToken)
    .finally(() => {
      refreshPromise = null;
    });
}

const accessToken = await refreshPromise;
```

이 구조의 의미는 다음과 같다.

```txt
진행 중인 refresh가 없으면
→ 내가 /auth/refresh 요청 시작

진행 중인 refresh가 이미 있으면
→ 새 refresh 요청을 보내지 않고 기존 refreshPromise를 await
```

즉, 동시에 401이 여러 개 터져도 실제 refresh 요청은 하나만 나간다.

```txt
/books  → 401 → refreshPromise 생성
/users  → 401 → 기존 refreshPromise await
/orders → 401 → 기존 refreshPromise await
```

refresh가 성공하면:

```txt
새 accessToken 반환
→ 기다리던 요청들이 모두 같은 accessToken을 받음
→ 각자 원래 요청을 재시도
```

## 4. 왜 가능한가

`refreshPromise`가 interceptor 함수 밖, 즉 모듈 스코프에 있기 때문이다.

```ts
let refreshPromise: Promise<string> | null = null;

api.interceptors.response.use(...);
```

그래서 interceptor가 여러 번 실행되어도 같은 `refreshPromise` 변수를 공유한다.

반대로 interceptor 내부에 두면 매번 새로 만들어져서 의미가 없다.

```ts
api.interceptors.response.use(async (error) => {
  let refreshPromise: Promise<string> | null = null;
  // 이러면 매 실행마다 null이라 공유 안 됨
});
```

또 JavaScript는 한 탭 안에서 싱글스레드 이벤트 루프 기반으로 동작하므로, 첫 번째 요청이 아래 코드를 실행하는 동기 구간 중간에 다른 요청이 끼어들지 않는다.

```ts
refreshPromise = api.post("/auth/refresh");
```

흐름은 다음과 같다.

```txt
A interceptor 실행
→ refreshPromise 없음
→ Promise 생성 후 할당
→ await에서 멈춤

B interceptor 실행
→ refreshPromise 있음
→ 기존 Promise await
```

## 5. 최종 interceptor 구조

```ts
import axios, { InternalAxiosRequestConfig } from "axios";
import { useAuthStore } from "@/stores/authStore";
import { api } from "@/lib/api";

type RetryRequestConfig = InternalAxiosRequestConfig & {
  _retry?: boolean;
};

let refreshPromise: Promise<string> | null = null;

api.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (!axios.isAxiosError(error)) {
      return Promise.reject(error);
    }

    const originalRequest = error.config as RetryRequestConfig | undefined;

    if (!originalRequest) {
      return Promise.reject(error);
    }

    if (error.response?.status !== 401) {
      return Promise.reject(error);
    }

    // refresh 요청 자체가 401이면 다시 refresh하지 않음
    if (originalRequest.url === "/auth/refresh") {
      useAuthStore.setState({ accessToken: undefined });
      return Promise.reject(error);
    }

    // 이미 재시도한 요청이 또 401이면 포기
    if (originalRequest._retry) {
      useAuthStore.setState({ accessToken: undefined });
      return Promise.reject(error);
    }

    originalRequest._retry = true;

    try {
      if (!refreshPromise) {
        refreshPromise = api
          .post("/auth/refresh")
          .then((res) => res.data.accessToken)
          .finally(() => {
            refreshPromise = null;
          });
      }

      const accessToken = await refreshPromise;

      useAuthStore.setState({ accessToken });

      originalRequest.headers.Authorization = `Bearer ${accessToken}`;

      return api(originalRequest);
    } catch (refreshError) {
      useAuthStore.setState({ accessToken: undefined });
      return Promise.reject(refreshError);
    }
  },
);
```

## 6. 같이 발견한 실수들

### 6.1. refresh 요청도 같은 interceptor를 타서 무한 루프 발생

처음에는 이렇게 되어 있었다.

```ts
await api.post("/auth/refresh");
```

그런데 `/auth/refresh`도 `api` 인스턴스를 쓰기 때문에, refresh가 401이면 interceptor가 다시 refresh를 호출하면서 무한 루프가 발생했다.

그래서 다음 가드가 필요했다.

```ts
if (originalRequest.url === "/auth/refresh") {
  useAuthStore.setState({ accessToken: undefined });
  return Promise.reject(error);
}
```

---

### 6.2. `_retry`는 axios 기본 타입에 없음

axios config 타입에는 `_retry`가 없으므로 그냥 쓰면 TypeScript가 거부한다.

```ts
originalRequest._retry = true;
```

해결 방법:

```ts
import type { InternalAxiosRequestConfig } from "axios";

type RetryRequestConfig = InternalAxiosRequestConfig & {
  _retry?: boolean;
};
```

그리고 interceptor에서 다음처럼 캐스팅한다.

```ts
const originalRequest = error.config as RetryRequestConfig | undefined;
```

---

### 6.3. React Query error는 사실상 unknown처럼 다뤄야 함

React Query의 `error`에는 뭐든 들어올 수 있다.

```ts
throw new Error("message");
throw "string";
throw { message: "error" };
throw axiosError;
```

그래서 전역 retry에서 무작정 다음처럼 접근하면 위험하다.

```ts
error.message;
```

axios를 쓴다면 다음처럼 좁혀서 처리하는 게 안전하다.

```ts
retry: (failureCount, error) => {
  if (axios.isAxiosError(error)) {
    const status = error.response?.status;

    if (status === 401 || status === 403) {
      return false;
    }
  }

  return failureCount < 2;
};
```

## 7. React Query와의 역할 분리

이번 결론은 다음 구조가 가장 깔끔하다.

```txt
axios interceptor
- accessToken 만료 401 처리
- refresh 성공 시 원 요청 재시도
- refresh 실패 시 accessToken 제거

Zustand
- accessToken 저장
- 인증 상태 관리

React Query
- 서버 상태 캐싱
- retry 정책
- /me 같은 유저 정보 조회

Route/Auth Guard
- accessToken 없으면 로그인 페이지 이동
```

React Query 전역 `onError`에서 다음과 같은 식으로 처리하는 것은 정상 플로우라기보다는 보험에 가깝다.

```txt
Axios interceptor에서도 해결 못하고 넘어온 401 처리
```

정상적으로 interceptor가 모든 인증 요청을 담당한다면, accessToken 만료로 인한 401은 React Query까지 거의 새지 않는 게 맞다.

## 8. 최종 결론

이번 트러블의 본질은 다음과 같다.

```txt
refresh token rotation 환경에서
동시 401 요청이 각각 refresh를 호출하면
refresh token이 한 번만 유효하기 때문에 경쟁 상태가 발생한다.
```

해결은 다음과 같다.

```txt
진행 중인 refresh Promise를 모듈 스코프에 저장하고,
동시 401 요청들이 같은 Promise를 await하게 만든다.
```

한 줄 요약:

```txt
동시 401 + refresh token rotation = refreshPromise 공유 패턴이 필요하다.
```

## 9. 오류 재현 방법

해당 문제를 확인하려면, access token을 일부러 잘못된 값으로 바꾼 뒤 인증이 필요한 api를 여러개 동시에 호출해야한다.

예를 들어 Zustand에 저장된 accessToken을 강제로 invalid token으로 바꾼다.

```ts
useAuthStore.setState({
  accessToken: "invalid-access-token",
});
```

그 후 인증이 필요한 api를 동시에 여러번 호출한다.

```ts
await Promise.allSettled([
  api.get("/books"),
  api.get("/books"),
  api.get("/books"),
  api.get("/books"),
  api.get("/books"),
]);
```

버튼을 만들어 버그 재현을 시도할 수 있음.

```tsx
<Button
  onClick={async () => {
    useAuthStore.setState({
      accessToken: "invalid-access-token",
    });

    const results = await Promise.allSettled([
      api.get("/books"),
      api.get("/books"),
      api.get("/books"),
      api.get("/books"),
      api.get("/books"),
    ]);

    console.log(results);
  }}
>
  401 테스트
</Button>
```
