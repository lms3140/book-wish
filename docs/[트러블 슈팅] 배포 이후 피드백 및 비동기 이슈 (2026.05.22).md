# 발견된 문제

실제로 배포하게 되면서 친구의 반응을 보고 발견한 문제다.

1. 삭제가 없음.
2. 생성,수정이 바로 이뤄지지않음. (비동기 이슈로 해결함)
3. 애매한 UI 배치

# 삭제가 없다!

이걸 왜 빼먹었는지 모르겠다. 이런 상황때문에 기록이 중요한듯 하다. 깃허브에 꾸준히 올렸다면 이런 상황에서 왜 뺐는지 체크할 수 있었을듯.

# 생성, 수정이 바로 이뤄지지않는 상황!

## 어떤 상황이었냐면...

생성, 수정을 했을때 테이블이 갱신되지않는 문제가 있었다.

분명히..

`queryClient.invalidateQueries({ queryKey: bookKeys.bookList });`

이 코드를 통해 react query가 새로 불러와야한다. 하지만 내가 실수한 곳이 있었다.

```ts
const bookAddSubmit = async (data: BookFormType) => {
  try {
    const result = bookAddSchema.parse(data);
    if (!result) return;

    createBook(result); // <<<< 이 부분!! 여기에요 여기!!

    queryClient.invalidateQueries({
      queryKey: ["bookList"],
    });
    alert("등록성공");

    reset(init);
  } catch {
    // 실패처리
  }
};
```

저 함수는 비동기다. 왜 이걸 놓치고있었을까?

**비동기 함수가 끝나기 전에 새로 불러와봐야 똑같은값이다.**

운이 좋으면 가끔 될 수 있을지도 모르지만, 운으로 불러와지는건 rpg게임으로 충분하다.

## 해결방법

단순히 저 함수를 await로 처리하는것도 괜찮겠지만, react query를 쓰고있으니 `useMutation`으로 처리하는게 더 직관적이고 유지보수하기 쉬울꺼라 생각해 useMutation으로 처리하기로 했다.

```ts
const mutation = useMutation({
  mutationFn: createBook,
  onSuccess: () => {
    queryClient.invalidateQueries({
      queryKey: ["bookList"],
    });
    alert("등록성공");
    reset(init);
  },
  onError: (error) => {
    alert(error.message || "등록 중 오류가 발생했습니다.");
  },
});
```

이것이 수정한 결과! 이전 코드보다 확실히 직관적이다.

## 결과

해결 방법을 적용하니 try catch보다 더 직관적인 모습으로 바뀌었다. 성공과 예외처리가 꽤 직관적으로 변하는 모습이 꽤 인상적이다.

기존 try catch는 try 내에서 해야할 작업과 성공처리가 같이 포함되어있어야하는 반면, useMutation은 처리해야하는것, 성공, 실패가 나눠져 있어 꽤 직관적이라 생각했다.

# 애매한 UI 위치

## 왜 애매한 UI 위치가 문제야?

실제로 이 앱을 테스트한 친구가 이런말을 했다.

"내가 이 ui에 익숙하지 않아서 그런가봐! 익숙해지면 괜찮을거같아."

내가 생각했을때 이런상황은 좋은 시그널은 아니다. 좋은 ui는 물 흐르듯 원하는 작업을 마칠 수 있고, 따로 배우지 않아도 벌써 익숙한 느낌을 받을 수 있어야 한다고 늘 생각해왔다.

## 그래서 뭘 해야할까..

좋은 ui를 배우는 방법은 실제 서비스 되고있는 프론트엔드 페이지를 기준을 세워 분석한다면 꽤 괜찮은 안목을 갖출 수 있을것이라 생각한다. 선배님들이 답지를 잘 만들어주셨으니까.

그리고 어떻게 하면 자연스럽고 편하게 쓸 수 있을지 고민하는것도 잊으면 안되겠지!
