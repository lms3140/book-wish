## 클릭 구현

shadcn의 DataTable에 클릭되면 강조되는것을 구현하고 싶었다.
하지만 DataTable에는 그런 기능은 존재하지 않았다(내가 못찾은걸 수도 있지만, 일단 안내엔 없었다)
그래서 스스로 구현을 해야한다!!

### 어떻게 구현했냐면..

선택된 값은 이미 BookStore의 형태로 가지고 있다. 이걸 응용한다면 구현자체는 쉬울것이라 생각했다.
TableRow 컴포넌트의 className에 BookStore를 기준으로 삼항연산자 하나를 넣었다.

### 한계점?

내가 구현한 방식에는 하나의 문제가 있다. 렌더링할때마다 저 연산을 셀 하나당 한번씩 돌린다는 사실!!
그래서 실험해봤다. 얼마나 무리를 주는걸까!!

```tsx
function App() {
  const [number, setNumber] = useState(0);
  const selectedBook = useBookStore((state) => state.selectedBook);
  const setSelectedBook = useBookStore((state) => state.setSelectedBook);

  return (
    <div>
      <button onClick={() => setNumber((prev) => prev + 1)}>{number}</button>
      <h1>Book List</h1>
      <ul>
        {books.map((book) => (
          <li
            key={book.id + "das" + number} // number이 바뀔 때마다 key가 바뀌도록 해서 전체 리스트가 리렌더링되도록 함
            className={book.id === selectedBook.id ? "text-amber-400" : ""} // 이것의 유무가 렌더링 비용에 영향을 미치는지 확인
            onClick={() => setSelectedBook(book)}
          >
            <strong>{book.title}</strong> by {book.author} (
            {book.publicationYear})<p>{book.description}</p>
          </li>
        ))}
      </ul>
    </div>
  );
}

export default App;
```

이런식으로 실험을 해봤고.... 놀랍게도 차이가 없었다!
Profiler 검증결과 약 40회의 커밋동안 평균 1.78ms / 1.79ms로 서로 동일한 성능을 보였고, 오히려 역전하는 경우도 있었다. (역전의 경우, 이상치라 판단하고는 있다.)

물론 더 복잡해지면 모르겠지만, 이러한 삼항연산자 사용방식은 속도에 영향이 없다고 봐도 무방할듯하다.

## 아직도 불편하다고?

### Detail 을 제거하는것 만으로는 충분하지 않다.

Detail 상태를 제거해 중복된 상황을 제거했지만, 아직도 같은 평가를 받았다. (내가 익숙해지면 괜찮을듯? 이라는 평가)

추측하기로는 UI관점에서 불필요하거나 의도가 충분히 전달되지않은 요소가 인지 코스트를 발생시켜 학습을 요구하게 된듯하다.<br/>
muted 방식을 사용해 지금 panel의 상태가 어떤지 보여주는 것은 현재 프로젝트에서 좋지않은 선택이었던듯하다.

### muted는 왜 좋지않았나..

왜 좋지 않았을까? 애초에 눈에 보인다는것 자체가 문제라고 생각하고 있다.  
수정, 등록이 같이 보이는 그 상황 자체 말이다.
흐릿하게 처리해도 사용자는 그것이 당연하지 않을 수 있다.  
이게 눌린건가? 이 버튼들이 뭘 의미하는거지? 라는게 개발한 나는 당연할지 몰라도. 유저들은 아닐 수 있다.

주제와는 별개로, 버튼 옆에 패널의 상태를 보여주는 "등록" "수정" 이라는 타이틀을 걸었지만 눈에 특별히 띄지않는것도 한몫한듯하다.

### 어떻게 해결해보려고?

muted방식을 빼버리고, 조건부 렌더링 방식으로 버튼 하나만 남겨두는 방법을 채택했다.
구현 방식은 switch case로 렌더링 함수를 만들어 제공하는 방식이다! 그러면 버튼이 하나만 있어 해당 버튼만 집중할 수 있게된다.

하지만 이 방식은 내 경험에 기반한 단점이 하나 있다.<br/>단순히 (수정) (등록) 이런식으로 내버려두면 지금 이 상태가 수정이야? 등록이야? 정말 햇갈린다.<br/>
그렇게 된다면 인지 코스트를 줄이려는 내 원래 목적이 수포로 돌아가는것은 물론 인지 코스트를 기존보다 더 쉽게 늘려버릴수 있다.

그래서 내가 선택한것은 버튼 문구를 길게 쓰는것. 수정-> 수정으로 전환 / 등록 -> 등록으로 전환.

그리고 사이드패널의 타이틀도 상태에 맞게 색을 칠해줬다. 등록은 파란색, 수정은 주황색.

### 어떻게 되었나??

정말 놀랍게도 "익숙해지면 괜찮을듯?" 이라는 평가가 사라졌다.
어려운 문제도 아니고, 구현이 어려운것도 아닌데 이런 평가의 변화가 생긴것이 내심 놀라웠다.
이런 간단한 ui의 선택이 큰 변화를 줄 수 있다는점이.. 참 놀라웠다. 유저를 위한 ui 설계 라는게 이런것이었나?

### 한계점

타이틀 색칠하기에서 색을 하드코딩했다.<br/>별것 아니지만, shadcn의 방식을 조금 벗어났기때문에 일관성에서 문제가 발생할 여지가 있다.  
하지만 프로젝트의 크기가 더 커지지않는이상 이정도의 문제는 넘어갈 수 있다고 생각한다. 어디까지나 개인을 위한 프로젝트니까.
